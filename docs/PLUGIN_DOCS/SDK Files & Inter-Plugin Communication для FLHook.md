## SDK-файлы

В папке:

```text id="mq25ix"
src/sdk
```

находятся заголовочные (`.h`) и библиотечные (`.lib`) файлы, необходимые для разработки плагинов под FLHook.

Они образуют интерфейс, через который ты можешь:

- вызывать **оригинальные функции FLServer**
- использовать **функции, экспортируемые самим FLHook**
- строить взаимодействие между плагинами

---

## Важно при сборке

> [!IMPORTANT]
> Не забудь добавить нужные `.lib` файлы в зависимости линковщика (**Linker Dependencies**),
> иначе проект может просто не собраться или не найдёт нужные символы.

---

# `plugin.h`

Один из ключевых файлов SDK:

```text id="egjpyi"
plugin.h
```

Он содержит:

- `PLUGIN_RETURNCODE`
- базовые структуры плагинов
- интерфейс **межплагинного взаимодействия**
- определения для обмена сообщениями между DLL

---

# Межплагинное взаимодействие

FLHook позволяет плагинам **общаться друг с другом** через специальный механизм обмена сообщениями.

То есть один плагин может:

- отправить команду другому
- передать данные
- вызвать определённое действие в другом модуле

---

## Как это работает

Для этого используется функция, экспортируемая FLHook:

```cpp id="4vfh0r"
Plugin_Communication(...)
```

А принимающий плагин должен реализовать callback:

```cpp id="k2e6l6"
Plugin_Communication_CallBack
```

---

# Пример обработчика сообщений

```cpp id="cwazie"
__declspec(dllexport) void Plugin_Communication_CallBack(PLUGIN_MESSAGE msg, void* data)
{
	// this is the hooked plugin communication function
	// we now check, if the message is for us
	if(msg == DO_SOMETHING) {
		// the message is for us, now we know what the actual data is, so we do a reinterpret cast
		CUSTOM_STRUCT* incoming_data = reinterpret_cast<CUSTOM_STRUCT*>(data);

		// do something here with the received data & instruction
	}
	return;
}
```

---

# Что здесь происходит

## Параметры callback-функции

### `msg`

Тип сообщения / команда, которую отправил другой плагин

### `data`

Указатель на произвольные данные, переданные вместе с сообщением

---

# Работа с `data`

Так как `data` имеет тип:

```cpp id="yn9sz9"
void*
```

ты сам должен знать, **что именно туда положил отправитель**.

Поэтому обычно используется:

```cpp id="0w6esd"
reinterpret_cast
```

чтобы привести `void*` к нужной структуре.

---

## Пример

Если ты знаешь, что другой плагин передал:

```cpp id="et0m0h"
CUSTOM_STRUCT
```

то ты можешь получить данные так:

```cpp id="3gd0x3"
CUSTOM_STRUCT* incoming_data = reinterpret_cast<CUSTOM_STRUCT*>(data);
```

---

# Что можно передавать

> [!NOTE]
> `CUSTOM_STRUCT` может быть **любой структурой любого размера**

То есть можно передавать:

- ID игрока
- имя персонажа
- параметры действия
- координаты
- флаги состояния
- любые свои внутренние данные

Главное, чтобы **обе стороны понимали одинаковую структуру**.

---

# Важное условие

> [!CAUTION]
> Функция:

```cpp id="07uh6d"
Plugin_Communication_CallBack
```

должна быть зарегистрирована как хук,
иначе FLHook **никогда её не вызовет**.

---

## Для старой схемы

Раньше это нужно было указывать в `[Hooks]`.

---

## Для новой схемы (`Get_PluginInfo`)

Нужно зарегистрировать её в списке хуков плагина.

Пример:

```cpp id="80m79h"
p_PI->mapHooks.insert(pair<string, int>("Plugin_Communication_CallBack", 0));
```

или в новой структуре хуков — если ты используешь современный стиль.

---

# Как отправить сообщение другому плагину

Отправляющий плагин делает примерно так:

```cpp id="os7itv"
CUSTOM_STRUCT outgoing_data;
Plugin_Communication(DO_SOMETHING, &outgoing_data);
```

---

# Что здесь происходит

1. Создаётся структура с данными
2. Вызывается:

```cpp id="a1bgfy"
Plugin_Communication(...)
```

3. Все плагины, которые подписаны на `Plugin_Communication_CallBack`, получают это сообщение
4. Каждый из них сам решает:
   - игнорировать сообщение
   - или обработать его

---

# Практическое применение

Этот механизм полезен, если ты хочешь разделить логику между несколькими плагинами.

Например:

- один плагин отслеживает события
- второй хранит состояние
- третий управляет UI / сообщениями / механикой

И они могут обмениваться данными **без жёсткой связки через прямые вызовы DLL-функций**.

---

# Пример сценария

## Плагин A

Обнаружил событие:

- игрок состыковался
- игрок включил cloak
- база изменила состояние

Он формирует структуру и шлёт сообщение:

```cpp id="1lvk9l"
Plugin_Communication(DO_SOMETHING, &outgoing_data);
```

---

## Плагин B

Получает это в:

```cpp id="2a49qm"
Plugin_Communication_CallBack(...)
```

и выполняет нужное действие.

---

# Преимущества такого подхода

- меньше жёсткой зависимости между DLL
- проще разделять функциональность
- легче строить модульную архитектуру
- удобнее расширять FLHook-сборку

---

# Итог

Если коротко:

- `src/sdk` — это твоя база для разработки
- `plugin.h` — ключевой файл
- `Plugin_Communication` — способ общения между плагинами
- `Plugin_Communication_CallBack` — обработчик входящих сообщений
