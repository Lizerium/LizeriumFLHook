## Введение

Сначала кратко о том, **как вообще работает система плагинов FLHook**.

Каждый плагин представляет собой **DLL-библиотеку**, которая экспортирует определённые функции. Эти функции затем вызываются самим **FLHook**.

- [SDK Files & Inter-Plugin Communication для FLHook](<SDK Files & Inter-Plugin Communication для FLHook.md>)
- [Troubleshooting для FLHook Plugins](<Troubleshooting для FLHook Plugins.md>)
- [Available Hooks — Хуки FLHook](<Available Hooks — Хуки FLHook.md>)

### Как это работает

Например:

- игрок отправляет сообщение в чат
- внутри FLHook вызывается функция `SubmitChat`
- затем FLHook проверяет все загруженные плагины
- если какой-либо плагин экспортирует соответствующий хук, эта функция вызывается уже внутри плагина

То есть по сути схема такая:

1. **FLHook** перехватывает (`hook`) вызовы `flserver`
2. Затем FLHook передаёт события подключённым плагинам

# Информация о плагине

## Изменения начиная с FLHook 1.6.1

Начиная с версии:

```text
FLHook 1.6.1 plugin
```

плагину **больше не нужен отдельный `.ini` файл** для регистрации.

Теперь достаточно:

- поместить DLL плагина прямо в папку:

```text
flhook_plugins/
```

---

## Обязательная экспортируемая функция

Каждый плагин теперь должен экспортировать функцию:

```cpp
Get_PluginInfo
```

Она заменяет старую систему регистрации через ini-файлы.

---

# Пример `Get_PluginInfo`

Ниже пример стандартной реализации на основе `tempban.dll`:

```cpp
EXPORT PLUGIN_INFO* Get_PluginInfo()
{
	PLUGIN_INFO* p_PI = new PLUGIN_INFO();
	p_PI->sName = "TempBan Plugin by w0dk4";
	p_PI->sShortName = "tempban";
	p_PI->bMayPause = true;
	p_PI->bMayUnload = true;
	p_PI->mapHooks.insert(pair<string, int>("HkTimerCheckKick", 0));
	p_PI->mapHooks.insert(pair<string, int>("HkIServerImpl::Login", 0));
	p_PI->mapHooks.insert(pair<string, int>("Plugin_Communication_CallBack", 0));
	p_PI->mapHooks.insert(pair<string, int>("ExecuteCommandString_Callback", 0));
	p_PI->mapHooks.insert(pair<string, int>("CmdHelp_Callback", 0));

	return p_PI;
}
```

---

# Что делает `Get_PluginInfo`

Эта функция сообщает FLHook:

- название плагина
- короткое имя
- можно ли его ставить на паузу
- можно ли его выгружать
- какие хуки должен обрабатывать этот плагин

---

## Поля структуры `PLUGIN_INFO`

### `sName`

Полное название плагина

### `sShortName`

Короткое техническое имя плагина

### `bMayPause`

Можно ли ставить плагин на паузу

### `bMayUnload`

Можно ли выгружать плагин без остановки сервера

### `mapHooks`

Список функций-хуков, которые FLHook должен подключить к этому плагину

---

# Как работает `mapHooks`

Пример:

```cpp
p_PI->mapHooks.insert(pair<string, int>("HkTimerCheckKick", 0));
```

Это означает:

- плагин хочет подписаться на хук `HkTimerCheckKick`
- при его вызове FLHook должен вызвать функцию из этого DLL

> [!CAUTION]
> Если вы указываете функцию в `mapHooks`, то вы **обязаны экспортировать её в DLL**.
> Иначе плагин будет работать некорректно или вообще не загрузится.

---

# Приоритет вызова хуков

Второй параметр в `mapHooks` — это **приоритет**.

Пример:

```cpp
p_PI->mapHooks.insert(pair<string, int>("SubmitChat", 3));
```

### Что это значит

Если несколько плагинов подписаны на один и тот же хук, то порядок вызова определяется именно этим числом.

---

## Пример

Допустим:

- античит-плагин подписан на функцию с приоритетом `3`
- логирующий плагин подписан на ту же функцию с приоритетом `0`

Тогда:

1. сначала вызов получит **античит**
2. потом — **логирующий плагин**

---

## Практический смысл

Это важно, если:

- один плагин должен **блокировать событие раньше других**
- нужно, чтобы логирование происходило **после проверки**
- несколько плагинов влияют на одну и ту же механику

# Итог

Система плагинов FLHook работает довольно просто:

- FLHook ловит события из `flserver`
- затем раздаёт их плагинам
- плагины подписываются на нужные хуки через `Get_PluginInfo`

Это делает архитектуру FLHook достаточно гибкой и позволяет расширять сервер без правки самого ядра.
