## Поиск ошибок

Если ты хочешь понять, **почему твой плагин ведёт себя как кусок стекла в мясорубке**, первым делом смотри сюда:

```text id="96o94k"
./EXE/flhook_logs/flhook.log
```

Это основной лог FLHook, и именно туда обычно попадают:

- исключения в callback-функциях
- ошибки выполнения плагинов
- сбои в хуках
- другие проблемы при работе DLL

---

## Что искать в `flhook.log`

Особенно полезно проверять лог, если:

- команда не вызывается
- хук срабатывает не так, как ожидалось
- плагин не грузится
- сервер падает или начинает вести себя странно
- какая-то callback-функция выбрасывает exception

---

## Исключения в callback-функциях

> [!IMPORTANT]
> Если одна из твоих callback-функций выбрасывает исключение в FLHook,
> это обычно фиксируется в:

```text id="u3l4m8"
flhook.log
```

---

## Рекомендация

Лучше не полагаться только на это.

Рекомендуется реализовывать **собственную обработку исключений** внутри критичных участков кода, чтобы:

- получать более понятные ошибки
- логировать контекст
- не терять информацию о состоянии перед падением

---

# Важный момент: Runtime Library и STL

Вот здесь уже начинается **классическая C++-шная мразота**, которая может убить тебе часы жизни.

---

## Проблема

Если ты отлаживаешь плагины и используешь **STL-объекты**, которые **делятся между FLHook и твоим плагином**, могут начаться очень мерзкие краши.

Например:

- FLHook создал `std::string`
- твой плагин попытался её освободить
- и всё полетело к чертям

---

## Почему это происходит

Потому что **разные runtime-библиотеки = разная память / аллокаторы / поведение STL**

---

# Что обязательно нужно сделать

> [!CAUTION]
> Если твой плагин использует STL-классы, которые пересекаются с FLHook,
> ты **обязан** использовать ту же runtime library, что и FLHook.

---

## Правильная настройка для обычной сборки

В настройках проекта Visual C++ выбери:

```text id="r0lmnq"
Multi-threaded DLL (/MD)
```

Путь в настройках:

```text id="mxys9f"
Project Settings → Code Generation → Runtime Library
```

---

## Иначе ты получишь

- debug assertion crashes
- странные падения
- heap corruption
- повреждённую память
- очень "весёлую" отладку

---

# Debug-сборка

Если ты собираешь **FLHook в debug-режиме**, тогда и плагин должен использовать соответствующий runtime:

```text id="l6a3ha"
Multi-threaded Debug DLL (/MDd)
```

---

## Главное правило

> [!IMPORTANT]
> Какая бы runtime library ни использовалась,
> она **должна совпадать** с той, что использует FLHook.

---

# Простое правило, которое спасает нервы

## Release + Release

```text id="g7m3ne"
/MD
```

## Debug + Debug

```text id="k8m37p"
/MDd
```

## Debug + Release / Release + Debug

```text id="lq5x68"
Плохо. Очень плохо.
```

# Performance Timer

В FLHook есть полезная встроенная штука — **таймер производительности**.

Он измеряет, сколько времени тратится на выполнение различных плагинов.

---

## Что он делает

Performance Timer отслеживает:

- сколько времени выполняется каждый плагин
- какие хуки работают слишком долго
- где сервер может тормозить из-за твоего кода

---

## Когда это полезно

Особенно полезно, если твой плагин:

- часто вызывается
- работает в таймере
- обрабатывает много игроков
- использует тяжёлую логику
- взаимодействует с файлами / базой / расчётами

---

# Как работает логирование таймера

Если выполнение занимает **дольше заданного порога**, это записывается в лог.

Порог можно настроить в:

```ini id="1b7z9z"
flhook.ini
```

---

## Параметр

```ini id="vvj8qu"
TimerThreshold=
```

---

## Что это значит

Если время выполнения плагина превышает значение `TimerThreshold`, FLHook зафиксирует это как потенциальную проблему производительности.

---

# Практическая польза

Это помогает находить:

- тяжёлые хуки
- плохие циклы
- лишние проверки
- лагучие команды
- неудачные таймеры
- проблемные места в многоплагинной сборке

---

# Что делать, если сервер лагает

Если сервер начинает:

- подвисать
- отвечать с задержкой
- тормозить на командах
- нестабильно работать под нагрузкой

то алгоритм такой:

---

## Чеклист

### 1. Проверить:

```text id="sl2e83"
flhook.log
```

### 2. Проверить:

```text id="1d7quh"
performance timer / perftimer logs
```

### 3. Проверить:

- не блокирует ли что-то главный поток
- нет ли тяжёлых STL-операций
- нет ли лишних проходов по игрокам / объектам
- не делаешь ли ты лишние файловые операции в callback-хуках

---

# Итог

Если коротко:

- **ошибки** смотри в `flhook.log`
- **совместимость памяти** обеспечивай через одинаковый runtime
- **производительность** отслеживай через Performance Timer

---

# Практический вывод

Если ты пишешь плагины для FLHook, то твой стандартный набор выживания — это:

- нормальные try/catch
- совпадающий runtime
- логирование
- контроль производительности

Без этого рано или поздно начнётся **типичный C++ серверный сатанизм**.
